## 🎯 第一部分：项目理解 (基础问题)

### Q1: 这个项目是做什么的？用一句话概括。

**标准答案：**
```
这是一个机器人与人类交互中的结构化偏好学习系统，
核心是训练一个reward model来评估轨迹的安全性和舒适性，
进而通过RL策略学习优化机器人的递物行为。
```

**展开说明（面试官追问时用）：**
```
背景：
  - 传统递物只优化成功率/避碰/效率，忽视人类的主观体验
  - 问题：难以衡量"这个递物是否真的安全、舒适、可接受"

我们的解决方案：
  1. 重构问题：从"递物成功"→"候选轨迹排序"（偏好学习）
  2. 设计系统：5层网络预测安全/舒适分数
  3. 分层Reward：安全优先(Gating) + 舒适优化(in safety region)
  4. 完整管道：从合成数据生成→模型训练→RL策略学习
```

### Q2: 为什么重构为"偏好排序问题"而不是直接预测轨迹？

**核心回答：**
```
难度对比：

❌ 直接预测轨迹（难）：
   输出空间巨大（时间序列 × 维度数）
   → 学习困难，泛化性差

✅ 从候选中排序（易）：
   输入：context + 3-5条候选
   输出：一个数字（排序得分）
   → 学习简单，泛化性好
```

**类比解释：**
```
直接预测轨迹 = 让学生写作文（开放式）→ 很难评估
偏好排序 = 让学生选择"A文章好于B文章"（多选题）→ 容易理解

在实践中：
  - 规划器本来就生成多个候选
  - 只需学会如何比较和排序
  - 符合人类的评价方式（A更安全 vs B更舒适）
```

### Q3: 项目的创新点是什么？

**三大创新：**

```
1️⃣ 分层Reward设计（Safety Gating + Comfort Optimization）
   创新点：不是平等加权的多目标优化
   
   传统方法（不行）：
      R = 0.5*safety + 0.5*comfort
      问题：不安全但舒适的轨迹也可能得高分
   
   我们的方法（对）：
      R = comfort - λ * ReLU(τ - safety)²
      
      含义：
      ├─ safety ≥ τ：只看comfort（正常竞争）
      ├─ safety < τ：被狠狠惩罚（-λ倍的平方项）
      └─ 结果：不安全永远得不到高分

   好处：符合人类心理（安全第一，再考虑舒适）


2️⃣ 几何参数模块（Interpretability）
   创新点：在黑盒神经网络中引入可解释性
   
   传统方法：end-to-end 黑盒，无法理解模型为什么这样评分
   
   我们的方法：
      Layer 2 = 显式计算几何特征
      ├─ Safety Ellipsoid：轨迹与人类的距离
      ├─ Comfort Fan：轨迹的平滑度、加速度
      └─ 这些特征既能输入神经网络，也能让人理解
   
   好处：
      ├─ 模型决策可解释
      ├─ 小样本学习效率高
      └─ 便于调试和验证


3️⃣ Candidate-First 的完整数据管道
   创新点：不是简单的"一对比较"，而是"全集管理"
   
   传统方法：AI标注提供孤立的A/B对比
   
   我们的方法：
      schema结构 = {
         context,           # 场景
         candidate_set,     # 完整的候选集（3-5条）
         training_pairs     # 从candidate_set中采样的对
      }
   
      质量控制4层：
      ├─ Schema Gate（格式检查）
      ├─ Geometry Consistency Gate（安全/舒适标签一致）
      ├─ Hierarchy Logic Gate（安全优先关系）
      └─ Coverage Gate（场景覆盖率）
   
   好处：
      ├─ 数据内部一致性强
      ├─ 自动检测标注错误
      └─ 支持后续的context-aware学习
```

## 🏗️ 第二部分：系统架构 (深度问题)

### Q4: 系统有哪5层？各层的作用是什么？

**完整答案：**

```
Layer 0: Candidate Generation （候选轨迹生成）
├─ 输入：context（场景）
├─ 输出：3-5条可行轨迹
├─ 来源：
│  ├─ classical：几何规划器（PRM、RRT等）
│  ├─ learned：神经网络策略
│  ├─ hybrid：两者结合
│  └─ safe_fallback：保底方案
├─ 为什么需要：从完整候选集中选择，比直接预测更稳定
└─ 代码位置：build_candidate_segments.py

                           ↓

Layer 1: Context + Trajectory Encoding （特征提取）
├─ Context编码：
│  ├─ 环境信息：地图、障碍物（用embedding）
│  ├─ 人类信息：位置、姿态、肢体特征
│  └─ 任务信息：递物目标位置
├─ Trajectory编码：
│  ├─ 输入：时间序列的位置/速度/加速度
│  ├─ 处理：RNN/Transformer 处理序列
│  └─ 输出：固定维度的向量
├─ 融合：Concatenate + 全连接层
└─ 输出：隐特征向量（比如256维）

                           ↓

Layer 2: Geometry Module （几何参数 - 可解释性）
├─ Safety Ellipsoid：
│  ├─ 定义：以人类身体为中心的椭球体
│  ├─ 计算：min(轨迹与椭球的距离) → 安全分 S(x)
│  ├─ 公式：S(x) = 1 - exp(-d²/σ²)  其中d是最小距离
│  └─ 范围：[0, 1]
├─ Comfort Fan：
│  ├─ 参数：速度、加速度、Jerk、光滑度
│  ├─ 计算：比较这些参数与"舒适范围"
│  ├─ 公式：C(x) = exp(-(v-v_opt)²/σ_v² - (a-a_opt)²/σ_a² - ...)
│  └─ 范围：[0, 1]
├─ 输入到Layer 3：[feat_layer1, S(x), C(x)]
└─ 可解释性：人类可以理解S和C是怎么算的

                           ↓

Layer 3: Backbone Fusion （骨干网络 - 特征融合）
├─ 输入：[Layer1特征, 几何特征]
├─ 网络：3-4层的全连接网络或Transformer
├─ 作用：融合所有信息，提取高阶特征
├─ 输出：隐表示（比如128维）
└─ 为什么需要：让模型学习Layer2没有显式编码的复杂关系

                           ↓

Layer 4: Multi-Head Supervision （分支头 - 多任务预测）
├─ Safety Score Head：
│  ├─ 结构：2-3层FC + Sigmoid
│  ├─ 输出：safety_score ∈ [0, 1]
│  ├─ 监督：MSE loss 与 score_target
│  └─ 作用：预测轨迹的安全性
│
├─ Comfort Score Head：
│  ├─ 结构：2-3层FC + Sigmoid
│  ├─ 输出：comfort_score ∈ [0, 1]
│  ├─ 监督：MSE loss 与 score_target
│  └─ 作用：预测轨迹的舒适性
│
├─ Reason Label Head（可选）：
│  ├─ 输出：{safety, comfort, mixed, unclear}
│  ├─ 监督：Cross-entropy loss
│  └─ 作用：预测为什么一个轨迹比另一个差
│
└─ Reaction Label Head（可选）：
   ├─ 输出：{hesitation, intervention, smooth}
   ├─ 监督：Cross-entropy loss
   └─ 作用：预测人类的反应

                           ↓

Layer 5: Structured Reward Output （最终输出）
├─ 计算：
│  ├─ Geometry Branch：
│  │  R_geom = C_geom - λ * ReLU(τ - S_geom)²
│  │
│  ├─ Learned Branch：
│  │  R_learned = comfort_score - λ * ReLU(τ - safety_score)²
│  │
│  └─ Final Fusion：
│     R_final = 0.8 * R_geom + 0.2 * R_learned
│
├─ 输出：
│  ├─ 主reward（用于排序）
│  ├─ 安全分（用于诊断）
│  ├─ 舒适分（用于诊断）
│  └─ 标签（原因、反应）
│
└─ 应用：用于候选轨迹排序或RL策略训练
```

### Q5: 为什么要有"几何参数模块"这一层？不能直接end-to-end吗？

**回答（分三个角度）：**

```
1️⃣ 可解释性
   ────────────
   几何模块的好处：
   ├─ 人类可以理解"为什么这个轨迹不安全"
   │  （因为离人类距离 < 0.3m）
   ├─ 便于系统调试和改进
   └─ 便于与领域专家沟通
   
   End-to-end的问题：
   ├─ 黑盒，无法解释
   └─ 难以找到问题所在

2️⃣ 数据效率（小样本学习）
   ────────────────────────
   几何模块的好处：
   ├─ 安全和舒适的定义是物理上的
   ├─ 用手工设计的特征，样本效率高
   ├─ 不需要大量数据就能学好
   └─ 比如：安全距离 = 某个物理常数，不需要数据验证
   
   End-to-end的问题：
   ├─ 需要大量数据才能学到这些物理规律
   └─ 在小样本场景下表现差

3️⃣ 可调试性和可控性
   ──────────────────
   几何模块的好处：
   ├─ 可以单独调整"安全阈值" τ
   ├─ 可以单独调整"舒适参数范围"
   ├─ 可以对特定场景微调
   └─ 比如：在人多的环境调高τ
   
   End-to-end的问题：
   ├─ 无法单独调整某个行为
   └─ 整个模型都要重新训练
```

**代码思路：**

```python
# Layer 2: Geometry Module

def compute_safety_score(trajectory, human_position):
    """计算轨迹的安全分"""
    # 输入：轨迹（时间序列的位置）、人类位置
    # 输出：安全分 ∈ [0, 1]
    
    # Step 1: 定义安全椭球体
    ellipsoid_params = {
        'center': human_position,  # 人类身体中心
        'semi_axes': [0.3, 0.3, 0.5]  # x, y, z方向的半径
    }
    
    # Step 2: 计算最小距离
    min_distance = 0
    for timestep_position in trajectory:
        # 计算这个位置到椭球体的距离
        dist_to_ellipsoid = distance_to_ellipsoid(
            timestep_position, 
            ellipsoid_params
        )
        min_distance = max(min_distance, dist_to_ellipsoid)
    
    # Step 3: 转换为安全分
    # d=0时 safety=0（碰撞），d→∞时 safety→1（完全安全）
    safety_score = 1 - exp(-(min_distance**2) / 0.1)
    
    return safety_score


def compute_comfort_score(trajectory, context):
    """计算轨迹的舒适分"""
    # 输入：轨迹、场景信息
    # 输出：舒适分 ∈ [0, 1]
    
    # Step 1: 提取轨迹特征
    velocities = compute_velocities(trajectory)
    accelerations = compute_accelerations(trajectory)
    jerks = compute_jerks(trajectory)
    smoothness = compute_smoothness(trajectory)
    
    # Step 2: 定义舒适范围
    comfort_ranges = {
        'velocity': (0.1, 0.5),      # 最舒适的速度范围
        'acceleration': (0.05, 0.3),  # 最舒适的加速度范围
        'jerk': (0, 0.2)               # 最舒适的Jerk范围
    }
    
    # Step 3: 计算偏离程度
    v_cost = distance_to_range(max(velocities), comfort_ranges['velocity'])
    a_cost = distance_to_range(max(accelerations), comfort_ranges['acceleration'])
    j_cost = distance_to_range(max(jerks), comfort_ranges['jerk'])
    
    # Step 4: 转换为舒适分
    total_cost = w_v * v_cost + w_a * a_cost + w_j * j_cost
    comfort_score = exp(-total_cost)
    
    return comfort_score
```

## 🎓 第三部分：核心算法 (技术问题)

### Q6: Reward公式 `R = comfort - λ * ReLU(τ - safety)²` 为什么这样设计？

**完整推导：**

```
问题：怎样设计一个同时考虑安全和舒适的reward？

❌ 方案1（加权和）：R = α*safety + β*comfort
   问题：
   ├─ 比如R = 0.5*0.1 + 0.5*0.95 = 0.525
   ├─ 即使超级不安全(0.1)，只要舒适(0.95)，总分也不错
   └─ 与直觉不符（不安全就是不安全）

❌ 方案2（乘积）：R = safety * comfort
   问题：
   ├─ 如果comfort=0.001（非常不舒适），即使safety=1也是0.001
   └─ 这样安全和舒适权重相同，与直觉也不符

✅ 方案3（我们的设计）：R = comfort - λ * ReLU(τ - safety)²
   
   设计思路：
   ┌─────────────────────────────────────────────┐
   │ 1️⃣ 基础是comfort（主要优化目标）             │
   │ 2️⃣ 如果safety足够（≥τ），就只看comfort      │
   │ 3️⃣ 如果safety不足（<τ），对comfort进行惩罚  │
   └─────────────────────────────────────────────┘
   
   数学解析：
   ├─ 当 safety ≥ τ 时：
   │  └─ ReLU(τ - safety) = ReLU(负数) = 0
   │  └─ 惩罚项 = 0
   │  └─ R = comfort（直接看舒适度）
   │
   ├─ 当 safety < τ 时：
   │  ├─ ReLU(τ - safety) = τ - safety > 0
   │  ├─ 惩罚项 = λ * (τ - safety)²
   │  ├─ R = comfort - 大正数（很低的reward）
   │  └─ 差距越大，惩罚越重（平方项）
   │
   └─ 极端情况：
      ├─ safety=0, comfort=1：R ≈ 1 - 10*0.5² = -1.5（很差）
      ├─ safety=0.8, comfort=0.3：R ≈ 0.3 - 0 = 0.3（还行）
      └─ 体现了"安全优先"的原则
```

**具体数值示例：**

```
参数设定：λ = 10, τ = 0.5

Case 1: 危险但舒适的轨迹
├─ safety_score = 0.2（靠人类太近）
├─ comfort_score = 0.9（速度平稳）
├─ R = 0.9 - 10 * (0.5 - 0.2)²
├─ R = 0.9 - 10 * 0.09
├─ R = 0.9 - 0.9 = 0  ← 被完全抵消了！
└─ 尽管舒适度高，但不安全所以得分很低

Case 2: 安全且舒适的轨迹
├─ safety_score = 0.8（离人类远）
├─ comfort_score = 0.7（加速度合理）
├─ R = 0.7 - 10 * ReLU(0.5 - 0.8)²
├─ R = 0.7 - 10 * ReLU(-0.3)²
├─ R = 0.7 - 10 * 0²  ← 因为ReLU负数返回0
├─ R = 0.7  ← 正常分数
└─ 既安全又舒适，得分就好

Case 3: 安全但不舒适的轨迹
├─ safety_score = 0.8
├─ comfort_score = 0.2（急加速、急转向）
├─ R = 0.2 - 0 = 0.2  ← 虽然不舒适，但没被惩罚
└─ 因为这比"危险"更容易接受
```

**为什么用平方而不是一次方？**

```
对比：
├─ 一次方：R = comfort - λ * ReLU(τ - safety)
│  └─ safety=0.3时：R = comfort - 10*0.2 = comfort - 2
│  └─ safety=0.1时：R = comfort - 10*0.4 = comfort - 4
│  └─ 差异不大（从-2到-4）
│
└─ 二次方：R = comfort - λ * ReLU(τ - safety)²
   └─ safety=0.3时：R = comfort - 10*0.04 = comfort - 0.4
   └─ safety=0.1时：R = comfort - 10*0.16 = comfort - 1.6
   └─ 差异大（从-0.4到-1.6）
   └─ 更有效地阻止选择不安全的轨迹

结论：平方项能更陡峭地惩罚不安全，符合"安全不可妥协"的原则
```

### Q7: 怎样从"打分"转换为"排序"进行训练？

**关键技术：Bradley-Terry Model**

```
问题：
├─ 我们有多条轨迹和它们的分数
├─ 但训练数据只给了我们"A优于B"的信息
└─ 怎样把这个对比关系变成loss function？

解决方案：
├─ 假设轨迹i的真实reward是 R_i
├─ 给定两条轨迹，概率 P(A > B) 应该遵循Bradley-Terry模型：
│
│  P(A > B) = σ(R_A - R_B) = 1 / (1 + exp(-(R_A - R_B)))
│            └─ σ = sigmoid函数
│
├─ 如果数据标注说"A优于B"（label=1），那么：
│  loss = -log(P(A > B)) = -log(σ(R_A - R_B))
│
├─ 如果数据标注说"B优于A"（label=0），那么：
│  loss = -log(P(B > A)) = -log(1 - σ(R_A - R_B))
│
└─ 两种情况可以统一为：
   loss = BCE(σ(R_A - R_B), label)
   其中 BCE = Binary Cross Entropy


具体计算示例：
───────────────

Case A: R_A = 0.8, R_B = 0.5, label = 1 (A确实优于B)
├─ σ(0.8 - 0.5) = σ(0.3) ≈ 0.574
├─ loss = -log(0.574) ≈ 0.555
├─ 解释：模型有57.4%的把握认为A优于B
│        但应该100%把握，所以loss较大
└─ 通过反向传播，模型会学会拉大R_A和R_B的差距

Case B: R_A = 0.9, R_B = 0.2, label = 1 (A确实优于B)  
├─ σ(0.9 - 0.2) = σ(0.7) ≈ 0.668
├─ loss = -log(0.668) ≈ 0.403
├─ 解释：模型有66.8%的把握
│        比Case A更好，但还不够，所以loss较小
└─ 再继续优化...

Case C: R_A = 1.0, R_B = 0.0, label = 1 (A确实优于B)
├─ σ(1.0) ≈ 0.731
├─ loss = -log(0.731) ≈ 0.314
├─ 解释：差距越大，预测越准确，loss越小
└─ 理想情况（完全正确）loss→0
```

**代码思路：**

```python
def pairwise_preference_loss(model, context, cand_a, cand_b, label):
    """
    计算偏好对的loss
    
    Args:
        model: 模型
        context: 场景
        cand_a, cand_b: 两条候选轨迹
        label: 1表示A优于B，0表示B优于A
    
    Returns:
        loss: 损失值
    """
    
    # Step 1: 获取预测分数
    safety_a, comfort_a = model(context, cand_a)
    safety_b, comfort_b = model(context, cand_b)
    
    # Step 2: 计算结构化reward
    reward_a = comfort_a - lambda_val * F.relu(tau - safety_a)**2
    reward_b = comfort_b - lambda_val * F.relu(tau - safety_b)**2
    
    # Step 3: 计算偏好概率（Bradley-Terry）
    logits = reward_a - reward_b  # 差值
    prob_a_better = torch.sigmoid(logits)  # 转换为概率
    
    # Step 4: 计算二元交叉熵loss
    pref_loss = F.binary_cross_entropy(prob_a_better, torch.tensor(label))
    
    return pref_loss
```

### Q8: 怎样生成合成数据？为什么不直接用真实数据？

**合成数据的优势：**

```
┌──────────────────────────────────────────────┐
│           Synthetic vs Real Data              │
├──────────────────────────────────────────────┤
│ 特性           │ 合成数据  │ 真实数据        │
├────────────────┼──────────┼─────────────────┤
```

**生成流程（代码逻辑）：**

```python
def generate_synthetic_training_record(scene_type):
    """生成一条合成训练记录"""
    
    # Step 1: 生成场景（Context）
    context = {
        'scene_type': scene_type,  # 'doorway', 'kitchen', 'table'
        'human_position': sample_human_position(scene_type),
        'human_posture': sample_posture(),
        'robot_start': sample_robot_start(scene_type),
        'obstacles': generate_obstacles(scene_type),
        'environment_map': load_map(scene_type)
    }
    
    # Step 2: 生成候选轨迹
    candidates = []
    
    # 古典规划器生成
    classical_traj = classical_planner.plan(
        start=context['robot_start'],
        goal=context['human_hand_position'],
        obstacles=context['obstacles']
    )
    candidates.append({
        'id': 'cand_001',
        'source': 'classical',
        'trajectory': classical_traj
    })
    
    # 学习方法生成
    learned_traj = learned_policy.sample(context)
    candidates.append({
        'id': 'cand_002',
        'source': 'learned',
        'trajectory': learned_traj
    })
    
    # 混合方法生成
    hybrid_traj = hybrid_planner.plan(context)
    candidates.append({
        'id': 'cand_003',
        'source': 'hybrid',
        'trajectory': hybrid_traj
    })
    
    # Step 3: 计算每条轨迹的几何特征
    for cand in candidates:
        cand['safety_score'] = compute_safety_ellipsoid(
            cand['trajectory'], 
            context
        )
        cand['comfort_score'] = compute_comfort_fan(
            cand['trajectory'], 
            context
        )
    
    # Step 4: 根据reward排序
    def trajectory_reward(cand):
        s = cand['safety_score']
        c = cand['comfort_score']
        return c - 10 * max(0, 0.5 - s)**2
    
    candidates.sort(key=trajectory_reward, reverse=True)
    
    # Step 5: 生成偏好对
    pairs = []
    for i in range(len(candidates) - 1):
        pair = {
            'candidate_a_id': candidates[i]['id'],      # 更好的
            'candidate_b_id': candidates[i+1]['id'],    # 较差的
            'labels': {
                'overall_preference': 'A',
                'reason_label': determine_reason(
                    candidates[i],
                    candidates[i+1]
                ),
                'safety_better_label': 'A' if candidates[i]['safety_score'] > candidates[i+1]['safety_score'] else 'B',
                'comfort_better_label': 'A' if candidates[i]['comfort_score'] > candidates[i+1]['comfort_score'] else 'B',
                'safety_score_target': [
                    candidates[i]['safety_score'],
                    candidates[i+1]['safety_score']
                ],
                'comfort_score_target': [
                    candidates[i]['comfort_score'],
                    candidates[i+1]['comfort_score']
                ]
            }
        }
        pairs.append(pair)
    
    # Step 6: 打包成JSON记录
    record = {
        'context_id': f'ctx_v2_{scene_type}_{uuid.uuid4()}',
        'context': context,
        'candidate_set': candidates,
        'training_pairs': pairs,
        'annotator_id': 'synthetic',
        'notes': f'Automatically generated {scene_type} scenario'
    }
    
    return record
```

### Q9: 数据质量怎样保证？有4层检查（Gate）具体是什么？

**四层质量控制详解：**

```
质量检查工作流：
───────────────

  原始合成记录
        ↓
   ┌─────────────────────────────────────────┐
   │ Gate 1: Schema Gate （格式检查）        │
   │ ─ 检查必需字段是否存在                   │
   │ ─ 检查字段类型和格式                     │
   │ ─ 检查enum值是否合法                     │
   └─────────────────────────────────────────┘
        ↓ 通过
   ┌─────────────────────────────────────────┐
   │ Gate 2: Geometry Consistency Gate       │
   │ ─ 检查score标签与实际分数是否一致        │
   │ ─ 检查reason_label是否符合defect        │
   └─────────────────────────────────────────┘
        ↓ 通过
   ┌─────────────────────────────────────────┐
   │ Gate 3: Hierarchy Logic Gate            │
   │ ─ 检查overall_preference是否符合        │
   │   structured reward公式                 │
   │ ─ 检查reason_label是否反映真实差异      │
   └─────────────────────────────────────────┘
        ↓ 通过
   ┌─────────────────────────────────────────┐
   │ Gate 4: Coverage Gate （覆盖率检查）    │
   │ ─ 检查所有场景类型是否都有样本          │
   │ ─ 检查各种candidate_source的覆盖        │
   │ ─ 检查reason_label的多样性              │
   └─────────────────────────────────────────┘
        ↓ 通过
   ✅ 高质量训练数据


具体检查项：

1️⃣ Schema Gate
───────────────
检查项 = [
    ('context.scene_type' in record, "Missing scene_type"),
    ('context.human_position' in record, "Missing human_position"),
    (len(record['candidate_set']) >= 2, "Less than 2 candidates"),
    (len(record['training_pairs']) >= 1, "No training pairs"),
    (all(p in VALID_SOURCES for p in candidate_sources), 
     "Invalid candidate source"),
]

示例检查：
├─ ✓ context_id 存在且唯一
├─ ✓ context 有所有必需字段
├─ ✓ candidate_set 中每条轨迹有id、trajectory、scores
├─ ✗ 如果缺少 safety_score 则失败


2️⃣ Geometry Consistency Gate
────────────────────────────
检查目的：确保score和label描述的是同一个东西

规则：
├─ Rule 1: safety_better_label 与 safety_score_target 一致
│  ├─ 如果 label="A"，那么 score_target[0] > score_target[1]
│  ├─ 不一致 → FAIL
│  └─ 示例：
│     "safety_better_label": "A",
│     "safety_score_target": [0.8, 0.7]  ✓ 一致
│
├─ Rule 2: comfort_better_label 与 comfort_score_target 一致
│  └─ 逻辑同上
│
└─ Rule 3: reason_label 与分数差异一致
   ├─ 如果 reason_label="safety"
   │  那么 |safety_A - safety_B| > |comfort_A - comfort_B|
   ├─ 如果 reason_label="comfort"
   │  那么 |comfort_A - comfort_B| > |safety_A - safety_B|
   └─ 如果 reason_label="mixed"
      那么差异都比较大


3️⃣ Hierarchy Logic Gate
──────────────────────
检查目的：确保总体偏好符合分层reward的逻辑

规则：
├─ Rule 1: 安全性优先
│  ├─ 如果 safety_A < τ 但 safety_B ≥ τ
│  ├─ 那么必须 preference="B"，即使 comfort_A > comfort_B
│  └─ 示例：
│     safety: [0.3, 0.6], comfort: [0.9, 0.4]
│     预期：preference = "B"（安全优先）
│     如果标注"A"则失败
│
├─ Rule 2: 安全范围内比较舒适
│  ├─ 如果 safety_A ≥ τ 且 safety_B ≥ τ
│  ├─ 那么 preference 应该是 comfort 更高的那个
│  └─ 示例：
│     safety: [0.8, 0.7], comfort: [0.5, 0.9]
│     预期：preference = "B"（都安全，选更舒适的）
│
└─ Rule 3: overall_preference ∝ reward
   ├─ 计算每个候选的 reward = comfort - λ*ReLU(τ-safety)²
   ├─ 比较大小
   └─ preference 应该指向 reward 更高的那个


4️⃣ Coverage Gate
────────────────
检查数据集级别的覆盖率

┌─ Scene Coverage ─────────────────────────────┐
│ 所有定义的场景类型是否都有样本？              │
│ ├─ doorway:     50 samples ✓                 │
│ ├─ kitchen:     45 samples ✓                 │
│ ├─ table:       48 samples ✓                 │
│ └─ hallway:     0 samples ✗ FAIL             │
└──────────────────────────────────────────────┘

┌─ Candidate Source Coverage ──────────────────┐
│ 每种生成方法都有样本？                        │
│ ├─ classical:   80 samples ✓                 │
│ ├─ learned:     75 samples ✓                 │
│ ├─ hybrid:      72 samples ✓                 │
│ └─ safe_fallback: 30 samples ✓               │
└──────────────────────────────────────────────┘

┌─ Reason Label Coverage ──────────────────────┐
│ 各种原因都有足够代表？                        │
│ ├─ safety reason: 60 pairs ✓                 │
│ ├─ comfort reason: 55 pairs ✓                │
│ ├─ mixed reason:   50 pairs ✓                │
│ └─ unclear reason: 2 pairs ✗ FAIL (太少)     │
└──────────────────────────────────────────────┘

┌─ Failure Mode Coverage ──────────────────────┐
│ 各种失败模式都有？                            │
│ ├─ Collision risk: 20 pairs ✓                │

└──────────────────────────────────────────────┘
```
