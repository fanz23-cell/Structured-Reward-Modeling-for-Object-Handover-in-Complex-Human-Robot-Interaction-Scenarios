# Data Pipeline V2: Comfort/Safety Candidate-First Synthetic And Weakly-Real

## Why Synthetic Must Be Candidate-First

The V2 system is not defined as "the world gives us one A/B pair".

It is defined as:

- one context
- multiple feasible candidates
- pairwise comparisons sampled from the candidate set

So synthetic data must preserve:

- `context`
- `candidate_set`
- `training_pairs`

Otherwise the training data would no longer match the reranker / selector framing.

## Why Geometry Truth Must Be Programmatic

Comfort fan and safety ellipsoid targets should not be written by GPT as free-form judgments.

They should be generated by program rules because:

- geometric labels need internal consistency
- comfort/safety better labels must match the score targets
- loser-side reason/subreason should align with actual control values
- safety-to-comfort hierarchy must be enforced deterministically

Programmatic generation is the only reliable way to ensure those constraints hold across many samples.

## Roles Of Synthetic And Weakly-Real Data

### Synthetic

Synthetic data is used for:

- full-coverage label generation
- score-target supervision
- subreason supervision
- fixed/global/contextual geometry ablations
- sanity checks for structured reward logic

### Weakly-Real

Weakly-real data is used for:

- schema sanity checking
- candidate-first annotation workflow validation
- real-data phase-1 compatibility checks

Weakly-real data should require only the main pairwise labels and may omit score/subreason targets.

## Quality Gates

The plus pipeline uses four gate categories:

1. schema gates
- required fields present
- legal enums
- valid candidate references inside the same context

2. geometry consistency gates
- `comfort_better_label` matches comfort score ordering
- `safety_better_label` matches safety score ordering
- subreason labels match the largest violation controls

3. hierarchy logic gates
- `overall_preference` follows the safety-to-comfort structured score
- `reason_label` reflects the loser main defect
- `reaction_label` is mapped from the defect pattern, not sampled randomly

4. coverage gates
- all requested scene types appear
- all requested candidate sources appear
- comfort and safety failure modes appear
- reason/reaction label coverage is checked

## Final Train/Val Structure

Both synthetic and weakly-real records keep the same top-level shape:

- `context_id`
- `context`
- `candidate_set`
- `training_pairs`
- `annotator_id`
- `notes`

Synthetic records additionally include richer program-generated debug/control fields such as:

- `synthetic_scores`
- `synthetic_controls`
- `quality_debug`

Weakly-real records may omit score/subreason targets.
